# ビジネス要件書

**システム名**: 統合業務管理システム
**対象組織**: ブーストコンサルティング / SALT2
**バージョン**: 1.0
**作成日**: 2026-02-18

---

## 1. 背景

ブーストコンサルティング及び子会社 SALT2 では、人事・勤怠・プロジェクト・財務情報を Google スプレッドシートを中心に管理している。データが分散しているため、情報の一元把握が困難であり、月末の給与計算・請求書発行・損益集計などに多大な手作業が発生している。

本システムは、これらの業務を一元管理・自動化する Web アプリケーションとして構築する。

---

## 2. 課題

| # | 課題 | 影響 |
|---|------|------|
| 1 | スプレッドシートが属人化しており、担当者不在時に業務が止まる | 継続性リスク |
| 2 | 月末の勤怠集計・請求書作成に手作業が集中する | 作業工数の増大 |
| 3 | メンバーのスキル情報が整理されておらず、アサイン判断に時間がかかる | 生産性低下 |
| 4 | プロジェクトごとの損益がリアルタイムで把握できない | 経営判断の遅延 |
| 5 | 勤怠の打刻・確認が Slack や紙ベースで行われており、集計が煩雑 | 精度低下・工数増大 |

## 3. 対象ユーザー

### 3.1 利用組織

| 項目 | 内容 |
|------|------|
| 親会社 | ブーストコンサルティング |
| 子会社 | SALT2 |
| 管理対象人数 | 30〜100 名程度 |

### 3.2 ユーザーロール

| ロール | 対象者 | 主な利用シーン |
|--------|--------|----------------|
| 管理者 | 役員・経営層 | 全機能へのフルアクセス、メンバー管理、PL 確認 |
| マネージャー | PM・リーダー | 担当 PJ 管理、勤怠承認、PJ 別 PL 閲覧 |
| 社員 | 正社員 | 勤怠打刻、自身の情報編集、担当 PJ 閲覧 |
| インターン | 本採用・研修インターン・研修メンバー | 勤怠打刻、自身の情報編集 |

## 4. 提供価値

| 価値 | 詳細 |
|------|------|
| 業務自動化 | 月末の勤怠集計・給与計算・請求書生成を自動実行 |
| データ一元化 | 人事・スキル・PJ・勤怠・財務を単一システムで管理 |
| 可視化 | スキルマトリクス・PL・カレンダーでチーム状況を即時把握 |
| Slack 連携 | 打刻通知・請求書送付・月末確認依頼を Slack で自動化 |
| アクセス制御 | ロールベースの権限管理により情報漏洩リスクを低減 |

---

## 5. 成功指標（KPI）

| 指標 | 現状 | 目標 |
|------|------|------|
| 月末請求書作成工数 | 手作業 数時間/月 | 自動生成（確認のみ）で 90% 削減 |
| 勤怠集計工数 | 手作業 数時間/月 | 自動集計で 80% 削減 |
| スキル確認〜アサイン決定時間 | 数時間〜1 日 | スキルマトリクス参照で即日対応 |
| PL 把握リードタイム | 月次手集計 | リアルタイム閲覧 |
| システム稼働率 | - | 99.5% 以上 |

## 6. スコープ（やる / やらない）

### やること（全機能・フルスコープ）

| モジュール | 概要 |
|-----------|------|
| M1: メンバー管理 | プロフィール管理、ステータス管理、報酬設定、権限管理 |
| M2: スキル管理 | スキルカテゴリ管理、5 段階評価、スキルマトリクス |
| M3: プロジェクト管理 | PJ 登録、ポジション定義、アサイン管理、工数管理 |
| M4: 勤怠管理 | 勤務予定登録、出退勤打刻、全体カレンダー、Slack 通知 |
| M5: 請求書生成 | 月末締め処理、給与計算、Excel 請求書自動生成、Slack 送付 |
| M6: PL 管理 | PJ 別損益、全社サマリー、キャッシュフロー管理 |

### やらないこと

| 項目 | 理由 |
|------|------|
| 経理部門での支払い実行 | 請求書作成までが自動化対象。支払いは経理部門で行う |
| ネイティブモバイルアプリ | Web のレスポンシブ対応で代替（要否は後日検討） |
| 外部会計ソフトとの連携 | Phase 1〜5 のスコープ外 |
| 採用管理（ATS） | 本システムの対象外 |

## 7. 制約・前提

| 区分 | 内容 |
|------|------|
| 期限 | 段階的リリース（Phase 1〜5 各 1〜2 ヶ月）。詳細スケジュールは別途確定 |
| 体制 | 小規模チーム（1〜3 名程度）での開発を想定 |
| 認証 | OAuth 2.0 採用（Google / Slack）。メール＋パスワード認証は後日検討 |
| セキュリティ | HTTPS 必須、JWT 認証、銀行口座情報の暗号化保存、OWASP Top 10 対応 |
| 技術スタック | Next.js (TypeScript) + Tailwind CSS + shadcn/ui + PostgreSQL (Prisma ORM) |
| ホスティング | Vercel / AWS（CDN・サーバーレス） |
| パフォーマンス | ページ読み込み 3 秒以内、API 応答 200ms 以内（p95）、同時 100 ユーザー以上 |
| データ保持 | 最低 3 年間 |
| バックアップ | 日次自動バックアップ、RTO 4 時間以内 |
| 多会社対応 | ブーストコンサルティングと SALT2 のデータを同一システムで管理（分離方針は別途確定）|

## 8. 用語

| 用語 | 定義 |
|------|------|
| メンバー | システムに登録された個人（役員・社員・インターン等を含む総称） |
| ステータス | メンバーの雇用形態区分（役員 / 社員 / 本採用インターン / 研修インターン / 研修メンバー） |
| ロール | システム上の権限区分（管理者 / マネージャー / 社員 / インターン） |
| アサイン | メンバーをプロジェクトの特定ポジションに割り当てること |
| 工数 | プロジェクトへの月間想定稼働時間 |
| 打刻 | 出退勤時刻の記録操作 |
| 日報 | 出勤時の「今日やること」・退勤時の「今日やったこと／明日やること」の入力 |
| PL | Profit and Loss（損益計算）。プロジェクト単位の売上・コスト・粗利を管理 |
| キャッシュフロー | 月次の入金・支出の管理 |
| 月額固定メンバー | 役員・社員など月額報酬で働くメンバー |
| 時給制メンバー | インターン等、実働時間に応じて報酬が発生するメンバー |
| Slack 連携 | Slack Web API を通じた通知・DM 送信の仕組み |
| 請求書 | 時給制メンバーが月末に発行する Excel 形式の請求書 |
| スキルマトリクス | 全メンバーのスキルレベルを一覧表示する画面 |
